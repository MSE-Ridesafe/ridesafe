# Dev Diary — 25th June 2026

## Focus: Making rides read like trips (list polish + a journey timeline)

With addresses now stored on every ride, today was about *showing* them well. We reworked
what each row says in the rides list, and we built a proper "journey" card for the detail
screen — a little timeline from where a ride started to where it ended. Along the way we made
a point of building that card so it isn't stuck at exactly two stops, because we know we'll
want to merge rides into multi-stop trips later.

## Reworking the rides list

We changed what a list row leads with. Instead of a timestamp headline, each row now leads
with the **destination** — the end address — so scanning the list reads like a list of places
you went, not a list of clock times. The time range and duration moved down into the
supporting line, and we dropped the max-speed clutter from the row (it still lives in the
detail view). When a ride hasn't been geocoded yet, the row falls back to a neutral
"Unknown location" rather than showing nothing.

The grouping from before (one card per day, relative headers) stays; this was purely about the
content and hierarchy *inside* each row.

## A journey card for the detail view

The detail screen used to list From / To / Start / End as plain label-value rows. We replaced
that with a **journey card**: a vertical timeline with an origin marker (a ring) and a
destination marker (a filled pin) joined by a continuous line, each stop showing its address
and the time we were there. The timestamps sit in their own column to the left of the line, and
the total trip time sits at the bottom. We also split each address into two tiers — the street
(or place name) is prominent, and the ZIP + city sits quietly underneath.

One small Compose discovery worth noting: getting the connecting line to be exactly the right
length between two icons is fiddly if you hardcode heights. We let the row measure itself
(`IntrinsicSize.Min`) and drew the line as weighted segments above and below each icon, so the
icon stays centered and the line is continuous no matter how tall the addresses are. No magic
numbers.

## Built for more than two stops (trip-merging prep)

This is the part we did deliberately for the future. The journey card doesn't take a hardcoded
start and end — it takes a **list of stops** (`JourneyCard(stops: List<JourneyStop>, ...)`). It
works out the markers and connectors purely from position: the first stop is the origin, the
last is the destination pin, and anything in between becomes a waypoint, with the line drawn so
the whole chain stays connected. A `JourneyStop` is just an address + a time.

Right now we hand it the two stops we have (ride start and ride end). But when we get to
**merging rides into a single trip** — several legs recorded back-to-back — we'll just hand the
same card a longer list and it renders origin → waypoints → destination with no changes to the
card itself. We wanted that seam in place before the merge feature, not after.

## Addresses, displayed properly

To make the two-tier address display trustworthy, we stopped trying to *parse* the geocoder's
formatted line (splitting on commas guessed wrong as soon as we drove somewhere with a named
building, and the ZIP + city would go missing). Instead we now read the `Address` object's
**concrete fields** — street + house number, or the place/POI name when there's no street, and
ZIP + city separately. We store those two lines newline-separated so the display can split them
back exactly, and we ran a small migration to re-geocode the rides we'd already recorded into
the new format. Country and admin areas are simply never read, so they're dropped by design.

## The route map: an upgrade, and a glitch we didn't beat

We bumped **maps-compose from v6 to v8.3.0** and cleaned up the fallout. The upgrade deprecated
`rememberMarkerState`, so the start/end pins now use `rememberUpdatedMarkerState` — a drop-in
swap, since both markers are static.

The bigger change was making the route map *interactive*. Until now, tapping the lite-mode
snapshot on the detail screen handed you off to the Google Maps app. We didn't want to leave the
app, so tapping now opens our own **full-screen map** in a dialog: a live, gesture-driven map
kept in 2D (pan / zoom / rotate on, tilt off), with the map toolbar hidden so taps stay in-app
instead of launching Maps. We also took it edge-to-edge — dropping the top/bottom safe-area
insets (`decorFitsSystemWindows = false`) so the map fills behind the system bars, with the close
button re-applying the insets so it stays reachable.

## The glitch we chased (and parked)

Opening the full-screen map looked rough: it appeared at a default zoom and then visibly
**snapped** to fit the route a beat later. This is inherent to how the fitting works — the SDK's
`newLatLngBounds` needs the map's measured size, so the fit can only run *after* the map has laid
out and reported `onMapLoaded`. The first frame shows the un-fitted camera; the fit lands on the
next. Hence the jump.

We went back and forth trying to kill it, and tried a few things:

- **Seed the full-screen camera from the lite preview.** The card's preview map has already
  fitted the route, so we tried reusing its camera position as the full-screen map's *initial*
  position — open already framed, no re-fit. It didn't hold up: lite-mode maps don't reliably
  write the fitted camera back to the state, so the full-screen map inherited a near-default
  position centred on an endpoint instead of the whole route.
- **Compute the fit zoom ourselves, up front.** We re-implemented Google's "bounds zoom level"
  Mercator math to derive the fit zoom from the measured viewport, so the camera could open
  already framed without waiting for layout. The centre was right, but the zoom came out *too
  deep* — the route no longer fit in frame — and it regressed both the small and full-screen
  maps. Out-mathing the SDK's own fitter was the wrong call.

In the end we **reverted to the native behaviour**: let the SDK fit the route with
`newLatLngBounds` after layout, slight snap and all. The interactivity, 2D gestures, edge-to-edge
full-screen, and the marker-API fix all stay; only the cosmetic loading glitch remains. We're
logging it as a **known bug to fix in the polishing stage** rather than spending more time on a
clever workaround now.

## Decisions we made

- **Destination-first list.** The most useful thing about a past ride is where it took you, so
  that's the headline.
- **One card, many stops.** Cheaper to make the journey card general now than to rewrite it when
  trip-merging lands.
- **Read fields, don't parse strings.** Pulling concrete address fields is far more robust than
  reverse-engineering a localized formatted line.
- **Ship native map behaviour, park the glitch.** A correct-but-glitchy SDK fit beat a clever
  custom one that broke the framing. The loading snap is cosmetic and goes on the polish list.

## What works at this point

- Everything builds, and the unit tests pass (address splitting/short-form and the duration
  formatting).
- The list reads as destinations, and the detail journey card renders the start → end timeline
  with the two-tier addresses and the total time.

## Notes / things to keep an eye on

- We haven't eyeballed the journey card on a device with three or more stops yet — the
  two-stop case is what we exercise today, so the multi-stop spacing is worth a look once we
  actually feed it a merged trip.
- **Known bug:** the full-screen route map opens at a default zoom and snaps to fit the route a
  frame later. Native `newLatLngBounds` behaviour, deferred to the polishing stage — see "The
  glitch we chased (and parked)" above.
