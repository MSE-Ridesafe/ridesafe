# Dev Diary — 23rd June 2026

## Focus: Rides UI (list + detail) on top of the recording engine

The recording engine already lands a summary row per ride in Room and streams the full
GPS + motion samples to a per-ride gzip'd NDJSON file. Today was about finally *seeing*
that data: a list of recorded rides and a detail screen that draws the route on a map.

## What was added

**Rides list — `RidesScreen`**
- Material3, grouped by calendar day. Each day is one `surfaceBright` Card holding that
  day's rides as `ListItem`s (divider-separated); a relative header sits above each card
  (Today / Yesterday / localized weekday-date).
- Each row shows time range · duration · max speed · vehicle name, plus an empty state.
- **DB-summary only** — the list never opens a sample file, so it stays fast regardless of
  how many rides exist. Distance lives on the detail screen.

**Ride detail — `RideDetailScreen`**
- Google Maps (`maps-compose`, lite mode) route card: polyline over the recorded fixes,
  start/end markers, camera framed to the track bounds.
- Card-based layout: Summary (start / end / duration), Speed (max + avg), Distance.
- Distance and average speed are computed **live from the track file**, because the
  analysis pass (ANL-02) that would fill `Ride.distanceMeters` / `avgSpeedMps` doesn't
  exist yet — those columns are still null on disk.

**Plumbing**
- `RidesViewModel` (app-scoped, hoisted in `RidesafeApp` like the garage VM): combines the
  rides + vehicles flows to attach a vehicle name to each row, and reads a ride's track off
  the main thread.
- Extracted the GPS reader out of `RideRecordingEngine` (it was a private method) into a
  shared `RideTrack.kt`: `ridesDir()`, `readRideLocations()`, `trackDistanceMeters()`. The
  engine now delegates to it, removing the duplication.
- Lifted the garage screen's private `DetailCard`/`DetailRow` into
  `ui/components/DetailCard.kt` (now takes a `String` title) and reused it on both detail
  screens.
- Navigation: added `RideDetailRoute`, wired `ridesEntries(onOpen/onBack)` and the rides
  back stack, mirroring the garage list→detail flow.
- Formatting helpers: `formatSpeed` (km/h vs mph) and a new `TimeFormatter`
  (`rideDay`, `formatTimeOfDay`, `formatRideDateTime`, `formatDuration`, `formatDayHeader`).
  EN + DE strings for everything.

**Dependencies / config**
- Added `maps-compose` 6.4.1, the INTERNET permission, and the Maps API-key `meta-data`
  injected via a `${MAPS_API_KEY}` manifest placeholder.
- The key is read from a gitignored `secrets.properties` (with `local.properties` as a
  fallback). Android Studio never rewrites that file, sidestepping the "changes will be
  erased" boilerplate in `local.properties`.

## Findings & decisions

- **Google Maps is a deliberate exception to the strictly-on-device rule.** Maps fetches
  tiles over the network and needs an API key. The user chose it knowingly over an offline
  map (MapLibre/osmdroid). Recorded as a project memory so it doesn't get "fixed" later.
- **List vs. detail cost split.** Reading every sample file just to show a distance column in
  the list would mean N file reads on load; deferred distance to the detail view only.
- **`stickyHeader` doesn't resolve** under Compose BOM 2026.06.00, so day headers are plain
  `item`s (no pin-on-scroll). Cheap to revisit if the API reappears.
- **Lite-mode camera race.** Framing the route to `LatLngBounds` before the map has a size
  throws, so the bounds move runs in `onMapLoaded` behind a `runCatching` guard.
- **`secrets.properties` vs `local.defaults.properties`.** Google's docs point at
  `local.defaults.properties`, but that's a secrets-gradle-plugin convention (a *committed*
  defaults file). We're not using that plugin, so a dedicated gitignored `secrets.properties`
  is the cleaner fit.
- **Duration format** was iterated to `d / h / min` with seconds dropped, e.g. `5 min`,
  `1 h 05 min`, `2 d 03 h 04 min`; leading zero units omitted.

## What works at this point

- **Route rendering is verified on a real device with real rides** — the polyline,
  start/end markers, and camera-fit-to-bounds all draw correctly against actual recorded
  tracks (not just a clean build).
- **Automatic ride recording via Bluetooth works reliably.** Proven end-to-end on a real
  test device: connecting to a mapped vehicle's Bluetooth starts recording, disconnecting
  finalizes the ride, and the resulting rides show up and replay correctly in the new UI.
- `:app:assembleDebug` succeeds — Maps AAR links and the manifest merges cleanly.
- Unit tests green: `trackDistanceMeters` (distance summing) and `formatDuration`
  (whole-minute formatting, leading-zero-unit omission).
- No DB migration needed — the existing `Ride` row and `RideDao` already carry everything
  the screens read.

## Known issues

- **One of four test rides has a bad first GPS coordinate** — the opening fix was recorded
  ~3 km away from the actual starting point, then the track snaps back to the real route.
  The other three rides were clean. Cause is currently unknown (a stale/cached first fix
  from the fused provider is a suspect, but unconfirmed). Worth watching whether it's the
  first fix specifically and whether an accuracy threshold would filter it.

## Not yet verified / open

- Root-cause the ~3 km first-coordinate bug (see Known issues) — happened on 1 of 4 rides,
  reason unknown.
- Distance / avg speed remain computed-on-open; wiring the real analysis pass (ANL-02) is a
  later round.
- No delete-ride action, and no live distance in the list — both intentionally deferred.
