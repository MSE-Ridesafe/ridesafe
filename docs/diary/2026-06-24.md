# Dev Diary — 24th June 2026

## Focus: Turning ride coordinates into real addresses (reverse geocoding)

Yesterday we got rides on screen and the route on a map. Today we wanted the rides to read
like places, not numbers — "to Hauptstraße, Hamburg" instead of a pair of decimals. So we
went looking for a way to turn the first and last GPS fix of a ride into a human address.

## Our discovery: Android has a free, native geocoder

The nice surprise: we didn't need a paid API or a new dependency for this. Android ships a
native `android.location.Geocoder` that does reverse geocoding (coordinate → address) for
free. Under the hood it calls a Google backend, so it *does* need a network connection — it's
not an on-device lookup. We're fine with that here, but we wrote the code so it always degrades 
gracefully when there's no network.

A couple of things we learned wiring it up:
- On our `minSdk` (34) the old synchronous `getFromLocation(...)` is deprecated; the modern
  call is **asynchronous** and hands results back through a `GeocodeListener` callback. We
  wrapped that callback in a `suspend` function (`reverseGeocode()`) so the rest of the app can
  just `await` an address.
- We guard it with `Geocoder.isPresent()`, a 10-second timeout, and a catch-all that resolves
  to `null`. So offline, no backend, or a slow lookup all just mean "no address yet" rather
  than a crash or a hang.

## How it roughly works

1. **Store, don't recompute.** When a ride is finalized we already keep its first/last GPS fix
   on the summary row. We added two nullable columns next to those — `startAddress` and
   `endAddress` — and persist the geocoded text there (Room migration 5 -> 6). The point is that
   addresses become real data on the ride: they can be displayed, indexed, filtered and
   searched later, instead of being re-fetched every time we draw the list.
2. **Backfill on app open.** Geocoding lives in the UI/analysis layer, not in the recorder. The
   rides `ViewModel` runs one pass per launch (`RideDao.needingAddresses()`): for every ride
   that has coordinates but no stored address yet, it geocodes the endpoints and writes them
   back. Because the Room `Flow` is our source of truth, the list updates itself the moment the
   write lands. This one mechanism covers three cases at once — the rides we'd already recorded,
   brand-new rides, and rides that failed to geocode earlier (e.g. offline) get retried next
   launch.
3. **Show it.** We refactored the rides views to surface the new data: the **list** now leads
   each row with the destination (the end address) instead of just a timestamp, and the **detail**
   view shows the start and end addresses. The list keeps a shortened form so rows stay tidy.

## Decisions we made

- **Geocode + persist, not geocode-on-the-fly.** We explicitly wanted the addresses stored so
  they're searchable down the line, so we took the migration over a throwaway lookup.
- **Plain TEXT columns for now.** Searchable with a simple `LIKE` when we eventually build search;
  we deliberately skipped full-text search (FTS) until there's actually a search UI to justify it.
- **List reads the DB only.** Consistent with yesterday — the list never opens a sample file, and
  geocoding never blocks it; rows just fill in addresses as the backfill completes.

## What works at this point

- The whole thing builds, and addresses resolve on a real device for real recorded rides — the
  rides list reads as destinations now, which is exactly the "see your trips as places" feel we
  were after.
- The offline path behaves: no network just leaves the address `null` and the next launch tries
  again, with nothing crashing in between.

## Notes / things to keep an eye on

- The address *formatting* (how we turn a geocoder result into the short label we show) is the
  part most likely to need refinement as we record more varied destinations.
